\section{Evaluation}
\label{sec:evaluation}

The evaluation of \toolname addressed three research questions:

\begin{enumerate}
\item[RQ1] Is \toolname effective at finding \UnderInvalidation bugs?
\item[RQ2] Do minimization and clustering reduce review burden?
\item[RQ3] Does \toolname find novel under-invalidation bugs?
\end{enumerate}

We tested \toolname on Google Chrome (149.0.7827.115)
  and Mozilla Firefox (152.0)
  using Selenium (4.40.0) via
  chromedriver (149.0.7827.115)
 and geckodriver (0.37.0).
\toolname itself runs on Python (3.13.11).

Because no existing fuzzer specifically targets under-invalidation bugs,
 our evaluation focuses on \toolname’s efficiency
 and the quality of the results it produces. 
For RQ1, we fuzzed each browser for 60 minutes
  and compared the effects of
  \toolname's tweaks to generation of
  the \texttt{content-visibility} and \texttt{display} properties.
For RQ2, we performed a longer, 9-hour fuzzing run on Mozilla Firefox
  and examined the sizes of original, minimized, and clustered bugs.
For RQ3, we examined \toolname-generated bugs
  filed in the Google Chrome and Mozilla Firefox bug trackers.
RQ3 understates the impact of \toolname on Mozilla Firefox:
  after a pilot, the Mozilla developers chose to run \toolname
  in Grizzly, Mozilla's internal fuzzing tool-chain~\cite{grizzly},
  so as to detect bugs before they make it into
  a user-visible release.

\subsection{RQ1: Finding Under-Invalidation Reports}

RQ1 asks whether \toolname
  is effective at triggering under-invalidation behavior
  in Chrome and Firefox.
As a secondary goal, we also test \toolname's tweaks to
  the \texttt{content-visibility} and \texttt{display} properties,
  which are implemented as default property weights.
\Cref{fig:bug-detect} shows bug counts over time
  for all four tested configurations.
Every configuration found
  dozens of reports within 60 minutes:
  Chromium finds \ChromiumNoWeightsBugs{} reports without property weights
  and \ChromiumWeightsBugs{} reports with property weights,
  while Firefox finds \FirefoxNoWeightsBugs{}
  and \FirefoxWeightsBugs{} reports, respectively.
The large counts in each case demonstrate that
  \toolname reliably finds under-invalidation bugs.

\Cref{tab:rq1-results} shows numerical results
  for candidate generation speed,
  total bug count, and bug detection rate.
Overall, \toolname tested
  hundreds of bug candidates per minute in both browsers
  and typically found between one and two bugs per minute.
This allows for a fine-grained comparison of
  the effect of \toolname's default property weights.
In Firefox, weighting raises the candidate count
  from \FirefoxNoWeightsBugs{} to \FirefoxWeightsBugs{}
  and the bug count from \FirefoxNoWeightsRate{} to \FirefoxWeightsRate{},
  while throughput remains nearly unchanged
  (\FirefoxNoWeightsSpeed{} versus \FirefoxWeightsSpeed{} tests per minute).
In other words, the default weights allow \toolname
  to find more bugs in Firefox at no additional cost.
In Chromium,
  weighting increases throughput
  but does not improve discovery rate:
  the unweighted configuration produces more reports
  (\ChromiumNoWeightsBugs{} versus \ChromiumWeightsBugs{})
  and a higher report-triggering rate
  (\ChromiumNoWeightsRate{} versus \ChromiumWeightsRate). 
The difference is due to differences in how Chrome and Firefox
  implement the \texttt{content-visibility} property.
Firefox's implementation does not generate false positives,
  so generating this property or not has little effect on Firefox;
  meanwhile, the higher probability of \texttt{display}
  raises the total number of bugs found.
By contrast, Chrome's implementation of \texttt{content-visibility}
  does cause false positives, which causes \toolname
  to fruitlessly attempt to minimize the false positive,
  until it is minimized all the way to zero modified properties,
  at which point the bug is useless.
Disabling this property thus dramatically reduces
  the number of minimization steps needed,
  which explains the increased throughput,
  while also reducing false positives,
  explaining the reduced bug rate.

\subsection{RQ2: Reducing Developer Triage Work}

A bug is most useful when it is easy to understand and analyze. 
RQ2 asks whether minimization and clustering reduce the burden
  of understanding and analyzing the bug.
As a secondary goal, we examined
  the impact of minimization and clustering on bug discovery speed.
Both outcomes were measured during a 9-hour fuzzing run of Mozilla Firefox
  with \toolname's default weights.
\Cref{fig:rq2-style-elements} shows the size of the generated bugs,
  under three different conditions:
  without either minimization or clustering;
  after minimization but not clustering;
  and then after clustering, considering the template for each cluster.
Size is measured by the total number of CSS properties
  in the base styles of the resulting bugs,
  though similar results would be seen for counting
  elements or modified styles.
Minimization reduces the style-element volume in generated bugs
  by approximately \StyleElementOriginalToMinimizedFactor{} relative to the
  original reports.
Combining minimization with clustering reduces the volume by a further
  \StyleElementMinimizedToClusteredFactor{} relative to minimized reports.
Together, minimization and clustering produce an approximately
  \StyleElementOriginalToClusteredFactor{} reduction in style-element volume,
  substantially lowering the review burden for developers using \toolname.

\begin{figure}[!t]
    \centering
    \includegraphics[width=\linewidth]{figures/firefox_style_elements.png}
    \caption{Total CSS style elements in grouped Firefox reports before
    minimization, after minimization, and after clustering.}
    \label{fig:rq2-style-elements}
\end{figure}
\begin{figure}[!t]
    \centering
    \includegraphics[width=\linewidth]{figures/combined_minimization_and_clustering_time_over_time.png}
    \caption{Cumulative minimization and clustering time for Firefox sorting
    and no-sorting configurations.}
    \label{fig:rq2-firefox}
\end{figure}

The minimization-and-clustering pipeline
  accounted for \RuntimePercentMinClust\ of the 9-hour Firefox run's runtime.
However, as shown in \Cref{fig:rq2-firefox},
  clustering actually reduces the minimization-and-clustering time:
  the clustering configuration executes $\sim$130,640 test cases,
  compared with $\sim$126,210 without clustering,
  and spends $\sim$4,550 seconds on minimization and clustering
  rather than $\sim$5,890 seconds without clustering.
This is because,
  when a newly-generated bug matches the template
  of an existing cluster, \toolname can skip minimizing the new bug.
Consequently, clustering
  reduces cumulative post-processing time by $\sim$22.6\%
  and enables $\sim$4,430 additional tests to run. 

\subsection{RQ3: Finding Confirmed Bugs}

The reports found in RQ1 and RQ2 do not, by themselves,
  evidence of distinct root-cause bugs.
RQ3 therefore asks whether \toolname finds
  a diverse collection of bugs that can survive external review.

\end{figure*}

\begin{figure}[!t]

    \caption{Bug-group and single-bug counts by total detected reports in
    the Firefox sorting run.}
    \label{fig:rq3-firefox-diversity}
\end{figure}

\Cref{fig:rq3-bug-group-sizes} shows the final distribution of bug groups
  for the nine-hour Firefox run in RQ2;
  there are 37 singleton bugs collected into a single bar.
As is typical for fuzzers, the largest cluster is quite large,
  with 203 total bugs, roughly a third of all bugs found in this run,
  but the total number of bug clusters is large,
  showing that \toolname finds a diversity of different bugs.
\Cref{fig:rq3-firefox-diversity} shows how the number of bug groups
  and singleton bugs changes over time.
After a rapid increase over the first three hours of the run,
  the number of singleton bugs reaches a rough steady state,
  as does the total number of bug groups.
This is because, while \toolname continues to test new bug candidates,
  and while a substantial fraction
  continue to trigger an under-invalidation bug,
  the clustering algorithm largely assigns these bugs to existing clusters.
In a long enough run, we expect that \toolname would generate
  multiple of every kind of bug that it could generate,
  and the number of singleton bugs would eventually approach 0
  while the number of bug clusters would hit some maximum.%
\footnote{Perhaps roughly 80, based on the number of bug groups
  plus the number of singleton bugs at the end of the run.}

Grouping reports alone does not establish that the underlying defects are novel.
We thus reported \NumBugsNovel{} bugs, found by \toolname,
  to the Chrome and Firefox bug trackers;
  \Cref{tab:rq3-reported-bugs} thus lists each bug's
  tracker ID, title, and current status.
\textbf{
All \NumBugsNovel{} bugs have been accepted by the developers,
  and \NumBugsFixed{} of them have already been fixed,
}
  demonstrating that the reports are actionable.

In Chrome issue 40188093,
  setting a \texttt{grid} display mode and \texttt{max-content} height
  caused an element's dimensions to be computed incorrectly.
A Chromium developer traced the failure to
  a \texttt{column-width} interaction that fell back from
  the new grid implementation to the legacy grid-layout path, commenting,
  \textit{``This will basically be fixed once we move multi-col over.''}
The report was later closed as
  \textit{``Fixed by the new grid fragmentation.''}
This demonstrates \toolname's ability to find under-invalidation bugs
  in under-development browser components.

In Firefox issue 1733276,
  setting an \texttt{inline} display mode on a margin-bearing element
  caused incremental and full layout to wrap the same text differently.
A Mozilla developer confirmed that 
  \textit{``the discrepancy is clearly a bug''}
  and explained that the issue resulted from insufficient reflow
  in response to the style change.
The minimized test case contains whitespace
  that is necessary to reproduce the failure.
Triggering a full reflow by zooming makes the layouts agree; fix pending.

In Chrome issue 40174062,
  \toolname was able to crash the LayoutNG grid implementation
  by adding a fixed-position element as a child of a grid.
A Chrome tester confirmed being able to reproduce the issue
  on macOS, Linux, and Windows.
Developers fixed an incorrect assumption---%
  that grid containers are the containing block
  of their out-of-flow children---%
  and verified that the crash no longer occurred.

As a result of their experience with \toolname,
  \textbf{
    Mozilla began running \toolname directly
    as part of their internal fuzzing workflow
  },
  further demonstrating the importance
  of finding under-invalidation bugs.
