\section{Evaluation}

Our evaluation of \toolname is guided by these fundamental questions:

\noindent\textbf{RQ1: Is \toolname at triggering an
\UnderInvalidation bugs?}

\noindent\textbf{RQ2: Do minimization and clustering reduce report complexity?}

\noindent\textbf{RQ3: Does \toolname find confirmed under-invalidation
bugs?}

\textbf{Benchmarks:} Table~\ref{tab:benchmarks} summarizes the browser
targets evaluated by \toolname.  Our evaluation uses Chromium,
which exercises the Blink layout engine, and Firefox, which exercises the
Gecko layout engine.  \toolname uses ChromeDriver and GeckoDriver,
respectively.
\input{figures/tab_benchmarks}

\textbf{Experiment Setup:} Because no existing fuzzer specifically targets under-invalidation bugs,
 our evaluation focuses on \toolname’s efficiency and quality of the results it produces. 
 For RQ1, we run each browser with and without
property weights for 60 minutes.  We save several characteristics of these runs including speed(number of tests ran per minute), bugs found, rate of bug discovery, and the overall execution time. During the 60 minute window every minute a snapshot of these metrics were saved to accurately track the metrics across time. This data is represented in Fig.~\ref{fig:bug-detect}, which shows the bugs detected over execution time, and Table~\ref{tab:rq1-results}, which showcases metrics at the end of the 60 minute window.
For RQ2, the evaluations compares Firefox runs with sorting enabled and disabled over a nine hour execution. We
measure the number of css properties in the original reports, minimized reports and clustered reports, cumulative
post-processing time of minimization and clustering, and execution time. Measuring total css properties takes into account only the bugs that are within a bug group as to not inflate the total css properties of grouped or minimized bugs that were never grouped. For RQ3, we report the status of
confirmed bugs and use report groups and single
bugs to describe the diversity of the report repository. 

\subsection{RQ1: Finding Under-Invalidation Reports}

RQ1 establishes whether \toolname can repeatedly drive independent browser
engines into under-invalidation behavior.  Property weighting is a secondary
comparison: it biases generation toward selected CSS properties, while the
unweighted configuration samples from the same property space without that
bias.  Fig.~\ref{fig:bug-detect} shows that reports continue to
accumulate during the 60-minute window, and Table~\ref{tab:rq1-results}
distinguishes report count from throughput and report-triggering rate.

\textbf{Outcomes:} Every configuration finds dozens of reports within 60 minutes: Chromium finds
\ChromiumNoWeightsBugs{} reports without property weights and
\ChromiumWeightsBugs{} reports with property weights, while Firefox finds
\FirefoxNoWeightsBugs{} and \FirefoxWeightsBugs{} reports, respectively.  The nonzero rates in both Blink and Gecko show
that \toolname reaches the target behavior in two independent layout engines,
rather than depending on one browser-specific implementation.

The weighted comparison explains how configuration choice affects that
ability.  In Firefox, weighting raises the report count from
\FirefoxNoWeightsBugs{} to \FirefoxWeightsBugs{} and the report-triggering rate
from \FirefoxNoWeightsRate{} to \FirefoxWeightsRate{}, while throughput remains
nearly unchanged (\FirefoxNoWeightsSpeed{} versus \FirefoxWeightsSpeed{} tests per
minute).  In Chromium, weighting increases throughput but does not improve
discovery rate: the unweighted configuration produces more reports
(\ChromiumNoWeightsBugs{} versus \ChromiumWeightsBugs{}) and a higher
report-triggering rate (\ChromiumNoWeightsRate{} versus
\ChromiumWeightsRate).  Thus, higher throughput alone does not explain 
the usefulness of property weights depends on the browser engine and
the properties they emphasize.

\textbf{RQ1:} \toolname finds under-invalidation reports in both Chromium and
Firefox within one hour.  Property weights improves discovery rate, but their
benefit is browser-dependent rather than universal.
\end{minipage}
}
\end{center}

\subsection{RQ2: Reducing Developer Triage Work}

Finding a bug is most useful when it is easy to understand and analyze.  RQ2 asks
whether minimization and clustering reduce that burden, in addition it is important to examine the effeciency impacts.  
Minimization removes unnecessary CSS style
elements from a report.  Clustering recognizes repeated reports and retains a
shared rule and minimized representative, which can in turn be used to avoid full minimization for every
bug found.  We therefore evaluate two linked outcomes: how much smaller the
developer-facing reports become, and whether the extra processing reduces
fuzzing throughput.  Fig.~\ref{fig:rq2-style-elements} measures the former;
Fig.~\ref{fig:rq2-firefox} measures the latter.

\begin{figure*}[!t]
    \centering
    \includegraphics[width=\textwidth]{figures/firefox_style_elements.png}
    \caption{Total CSS style elements in grouped Firefox reports before
    minimization, after minimization, and after clustering.}
    \label{fig:rq2-style-elements}
\end{figure*}
\begin{figure*}[!t]
    \centering
    \includegraphics[width=\textwidth]{figures/combined_minimization_and_clustering_time_over_time.png}
    \caption{Cumulative minimization and clustering time for Firefox sorting
    and no-sorting configurations.}
    \label{fig:rq2-firefox}
\end{figure*}

\noindent\textbf{Outcomes:} The report-size results show that the clustering and minimization pipeline
meaningfully changes the volume of a bug analysis needed.  Minimization reduces the
style elements in grouped reports by \StyleElementOriginalToMinimizedReduction
relative to the original reports.  Combining minimization with clustering
reduces the style-element volume by
\StyleElementOriginalToClusteredReduction\ relative to originals and by
\StyleElementMinimizedToClusteredReduction\ relative to minimized reports.
Clustering removes a significant amount of repeated work across reports that
represent the same behavior. \toolname retains the group's shared rule and minimized
representative instead of independently minimizing and inspecting the new
report. Developers can therefore focus on one representative report per
behavior rather than repeatedly examining equivalent reports.

In addition the minimization-and-sorting pipeline accounts for only
\RuntimePercentMinClust\ of the Firefox sorting run's runtime.
Moreover, specifically the sorting of bug reports into bug groups does not introduce an overhead;
instead, it reduces post-processing cost when compared to simply minimizing each bug.
At the latest common snapshot, the sorting configuration executes
$\sim$130,640 test cases, compared with $\sim$126,210 without sorting, while
spending $\sim$4,550 seconds on minimization and sorting rather than
$\sim$5,890 seconds without sorting.  When a report matches an existing
group, \toolname can reuse that group's representative instead of fully
minimizing the new report.  Consequently, sorting reduces cumulative
post-processing time by $\sim$22.6\% and enables $\sim$4,430 additional tests
to run. 

\begin{center}
\fbox{
\begin{minipage}{0.95\linewidth}
\textbf{RQ2:} Minimization and clustering make reports substantially easier
to dissect while preserving, and in this experiment improving, fuzzing
throughput.  They reduce both the size of individual reports and the repeated
work spent on reports already represented by a bug group.
\end{minipage}
}
\end{center}

\subsection{RQ3: Finding Confirmed Bugs}

The candidate reports measured in RQ1 are not, by themselves, evidence of
distinct root-cause bugs.  RQ3 therefore asks whether \toolname produces
reports that survive external developer review.  \toolname produced
\NumBugsNovel{} confirmed under-invalidation bugs; Table~\ref{tab:rq3-reported-bugs} records their
current statuses.  \NumBugsFixed{} of these reports were fixed or closed by
developers.  We also reproduce \MozillaBugs{} human-submitted Mozilla bugs,
showing that the oracle can reach known under-invalidation behavior.

Clustering provides complementary evidence about report diversity before
manual validation.  Fig.~\ref{fig:rq3-firefox-diversity} shows how reports
are divided into bug groups and single bugs over time, while

\noindent\textbf{Outcomes:} Table~\ref{tab:rq3-reported-bugs} establishes
that \toolname's reports are actionable beyond the experimental oracle.  It
produced \NumBugsNovel{} confirmed bugs, and
\NumBugsFixed{} were fixed or closed.  These outcomes provide external evidence
that the tool reaches real under-invalidation defects rather than only
reporting test artifacts.

The clustering results provide evidence that \toolname discovers a diverse set
of potential bugs rather than merely generating many syntactic variations of
the same behavior. The
clustering process groups reports with shared behavior while retaining a
minimized representative for each group.  At the latest common snapshot, the
Firefox sorting run produces \FirefoxSortingReports{} reports across
\FirefoxSortingGroups{} groups and \FirefoxSortingSingleBugs{} unmatched reports.
The number of groups, each represented by distinct HTML
and CSS combinations, indicates that the report repository contains
substantial behavioral variation rather than a single bug expressed
repeatedly.  These counts are evidence of potential diversity, not confirmed
root-cause-bug counts; confirmation of novelty comes from the submitted
reports reviewed by browser developers in
Table~\ref{tab:rq3-reported-bugs}.
