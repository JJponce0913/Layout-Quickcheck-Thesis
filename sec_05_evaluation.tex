\section{EVALUATION}

Our evaluation of \toolname's browser fuzzing
capabilities is guided by the following fundamental questions:

\noindent\textbf{RQ1: How effective is \toolname at triggering an
\UnderInvalidation bugs?}

\noindent\textbf{RQ2: To what extent does sorting and clustering improve browser fuzzing?}

\noindent\textbf{RQ3: Is \toolname effective at finding new \UnderInvalidation bugs?}

\textbf{Benchmarks:} Table~\ref{tab:benchmarks} summarizes the
browser targets supported by \toolname.  Our quantitative evaluation uses
Chromium, which exercises the Blink layout engine, and Firefox, which
exercises the Gecko layout engine. \toolname uses ChromeDriver and GeckoDriver, respectively.\todo{SAFARI DRIVER?}
\input{figures/tab_benchmarks}

\textbf{Experiment Setup:} For RQ1, we run Chromium and Firefox with and
without property weights for 60 minutes.  We compare cumulative reports,
test-case throughput, and report-triggering rate.  For RQ2, we run
Firefox with sorting enabled and disabled over nine hours.
We measure cumulative minimization and sorting time
and the style elements retained after minimization and clustering.  For RQ3,
we inspect the resulting groups and single-bug reports, then track the status
of the novel bugs submitted to browser developers.  Group and single-bug
counts are clustering snapshots, not independently confirmed root-cause-bug
counts.

\subsection{RQ1: Bug Discovery and Property Weights}

RQ1 asks whether \toolname can effeciently find under-invalidation reports in
multiple browser engines and whether property weights improve that process.
Property weighting biases generation toward selected CSS properties; the
unweighted configuration samples from the same property space without that
bias.  Figure~\ref{fig:evaluation-overview} visualizes how reports accumulate
over the 60-minute run.  Table~\ref{tab:rq1-results} summarizes the
final throughput, report count, and report-triggering rate for each browser.

\textbf{Outcomes:} \toolname finds dozens of reports in both engines within

and table provide evidence that
weights can improve discovery, particularly in Firefox, but the effect is
browser-dependent.

\subsection{RQ2: Reducing Developer Triage Work}

RQ2 asks whether minimization and clustering make reports easier for a
developer to dissect without sacrificing fuzzing efficiency.
Minimization removes unnecessary CSS style elements from each report.
Clustering then identifies repeated reports and retains a shared rule and a
minimized representative instead of requiring full minimization for every
match.  Figure~\ref{fig:rq2-style-elements} shows the resulting reduction in
style elements for grouped Firefox reports.  Figure~\ref{fig:rq2-firefox}
shows the cumulative post-processing time for sorting and no-sorting runs.
\begin{figure}[t]
    \centering
    \includegraphics[width=\linewidth]{figures/firefox_style_elements.png}
    \caption{Total CSS style elements in grouped Firefox reports before
    minimization, after minimization, and after clustering.}
    \label{fig:rq2-style-elements}
\end{figure}
\begin{figure}[t]
    \centering
    \includegraphics[width=\linewidth]{figures/combined_minimization_and_clustering_time_over_time.png}
    \caption{Cumulative minimization and clustering time for Firefox sorting
    and no-sorting configurations.}
    \label{fig:rq2-firefox}
\end{figure}



\noindent\textbf{Outcomes:} Figure~\ref{fig:rq2-style-elements} shows that
minimization reduces style elements by
\StyleElementOriginalToMinimizedReduction\ relative to the original grouped
reports.  Combining minimization with clustering reduces the style-element
volume by \StyleElementOriginalToClusteredReduction\ relative to originals and
by \StyleElementMinimizedToClusteredReduction\ relative to minimized reports.
This reduction is achieved while minimization and sorting account for only
\RuntimePercentMinClust\ of the Firefox sorting run's runtime.

Figure~\ref{fig:rq2-firefox} shows lower cumulative post-processing time for
sorting.  At the latest common snapshot, the sorting configuration executes
130,642 test cases, compared with 126,212 without sorting, while accumulating
4,554 seconds of minimization and sorting time instead of 5,885
seconds.  Thus,
recognizing a report that matches an existing group avoids unnecessary full
minimization, reduces cumulative post-processing time by 22.6\%, and allows
4,430 additional tests to run.

\begin{center}
\fbox{
\begin{minipage}{0.95\linewidth}
\textbf{RQ2:} Minimization and clustering substantially reduce the CSS
properties developers must inspect.  The pipeline consumes a modest share of
runtime and can improve throughput by avoiding full minimization for reports
already represented by a bug group.
\end{minipage}
}
\end{center}

reports, 29 groups, and 91 single bugs.  The two clustering figures show that
the fuzzer repeatedly reaches related behaviors while retaining unmatched
reports for separate validation.  These group and single-bug counts remain
triage categories rather than independently confirmed root-cause counts.
